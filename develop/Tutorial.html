Part 1: Creating a Prefetcher¶

We will build a stride prefetcher from scratch. A stride prefetcher tracks the addresses accessed by each instruction pointer (IP). When it detects a repeating stride pattern, it prefetches ahead in that pattern.

Step 1: Create the Directory¶

Each module lives in its own directory under the interface directory it implements. Prefetchers go under prefetcher/:

mkdir prefetcher/my_stride

The Makefile automatically discovers .cc files under prefetcher/, replacement/, branch/, and btb/, so there is no build system configuration needed.

Step 2: Write the Header¶

Create prefetcher/my_stride/my_stride.h:

  1#ifndef MY_STRIDE_H
  2#define MY_STRIDE_H
  3
  4// <map> gives us std::map for our tracking table.
  5#include <map>
  6
  7// "address.h" provides champsim::address and champsim::block_number — ChampSim's
  8// type-safe address types.  Raw uint64_t addresses are error-prone (byte vs block
  9// vs page ambiguity), so ChampSim wraps them in distinct types.
 10#include "address.h"
 11
 12// "modules.h" is the central module-system header.  It provides:
 13//   - champsim::modules::prefetcher   (the interface we inherit from)
 14//   - champsim::modules::ModuleBuilder (the constructor parameter)
 15//   - champsim::modules::cache_module  (the parent type for prefetchers)
 16//   - register_module<T>  (used to register our implementation)
 17#include "modules.h"
 18
 19// --------------------------------------------------------------------------
 20// my_stride — A stride prefetcher with configurable degree.
 21//
 22// Inherits from champsim::modules::prefetcher, which itself inherits from
 23// module_base<prefetcher, cache_module>.  That template establishes two things:
 24//   1. Prefetchers are attached to a cache_module (the "parent").
 25//   2. The static registration/factory infrastructure (register_module,
 26//      create_instance) is scoped to the prefetcher interface.
 27// --------------------------------------------------------------------------
 28struct my_stride : public champsim::modules::prefetcher {
 29
 30    // --- Per-IP tracking state ---
 31    // We track each instruction pointer's last cache-line address and the
 32    // stride between the two most recent accesses.  When two consecutive
 33    // strides match, we have a pattern to prefetch.
 34    struct tracker_entry {
 35        champsim::block_number last_addr{};                 // cache-line address of last access
 36        champsim::block_number::difference_type stride{};   // last observed stride (in cache lines)
 37    };
 38
 39    // --- Configurable fields ---
 40    // "degree" controls how many cache lines ahead we prefetch once a stride
 41    // pattern is detected.  It defaults to 3 but can be overridden from JSON.
 42    int degree = 3;
 43
 44    // --- Parent cache pointer ---
 45    // We store a pointer to the parent cache_module so we can query runtime
 46    // state like MSHR occupancy.  This pointer is obtained from the
 47    // ModuleBuilder in the constructor.
 48    champsim::modules::cache_module* cache_ = nullptr;
 49
 50    // --- Tracking table ---
 51    // Maps instruction pointers to their tracker entries.  Using std::map
 52    // here for simplicity; a real design might use a fixed-size LRU table
 53    // (see champsim::msl::lru_table in ip_stride).
 54    std::map<champsim::address, tracker_entry> table;
 55
 56    // --- Constructor ---
 57    // Every module MUST accept exactly one argument: a ModuleBuilder.
 58    // The module system calls this constructor when it instantiates the
 59    // module, passing in a builder pre-populated with JSON parameters
 60    // and a reference to the parent module.
 61    my_stride(champsim::modules::ModuleBuilder builder);
 62
 63    // --- Interface overrides ---
 64    // champsim::modules::prefetcher declares six pure virtual methods.
 65    // We MUST override all of them.  For methods we don't need, we provide
 66    // an empty body with "override {}".
 67    //
 68    // prefetcher_initialize: Called once before simulation begins.
 69    void prefetcher_initialize() override {}
 70
 71    // prefetcher_cache_operate: Called on every cache access (hit or miss).
 72    // This is the main hook — it's where we detect strides and issue
 73    // prefetches.  Returns metadata to pass through to future callbacks.
 74    uint32_t prefetcher_cache_operate(champsim::address addr, champsim::address ip,
 75        bool cache_hit, bool useful_prefetch, access_type type,
 76        uint32_t metadata_in) override;
 77
 78    // prefetcher_cache_fill: Called when a cache line is filled (loaded into
 79    // the cache).  Useful for feedback-directed prefetching.  We don't need
 80    // it for a simple stride prefetcher, so we just pass through the metadata.
 81    uint32_t prefetcher_cache_fill(champsim::address addr, long set, long way,
 82        bool prefetch, champsim::address evicted_addr, uint32_t metadata_in) override
 83    { return metadata_in; }
 84
 85    // prefetcher_cycle_operate: Called once every cycle.  Useful if your
 86    // prefetcher needs to do work that isn't tied to a specific cache access
 87    // (e.g. draining a lookahead queue, as ip_stride does).
 88    void prefetcher_cycle_operate() override {}
 89
 90    // prefetcher_final_stats: Called at the end of simulation.  Use it to
 91    // print custom statistics.
 92    void prefetcher_final_stats() override {}
 93
 94    // prefetcher_branch_operate: Called on every branch instruction executed
 95    // by the core attached to this cache.  Some prefetchers (e.g. TAGE-based)
 96    // use branch outcomes.  We don't, so we stub it.
 97    void prefetcher_branch_operate(champsim::address ip, uint8_t branch_type,
 98        champsim::address branch_target) override {}
 99};
100
101#endif

Why inherit from champsim::modules::prefetcher? It is the interface that defines the contract between the cache and its prefetcher(s). Every cache expects its prefetcher submodules to implement these six methods. The module_base machinery behind prefetcher provides the static factory (create_instance, register_module) that the config system uses to instantiate your module by name.

Why must the constructor take ModuleBuilder? The registration macro stores a factory lambda [](ModuleBuilder b){ return std::make_unique<T>(b); }. If your constructor has a different signature, that lambda won’t compile and registration will fail.

Step 3: Implement the Module¶

Create prefetcher/my_stride/my_stride.cc:

  1#include "my_stride.h"
  2
  3// "cache.h" provides the concrete CACHE class.  We don't strictly need it here
  4// because we only interact through the cache_module interface, but some modules
  5// may need it for additional helpers.
  6#include "cache.h"
  7
  8// =========================================================================
  9// REGISTRATION
 10//
 11// This single line registers our class with the module system under the name
 12// "my_stride".  When a config file says "model": "my_stride", the factory
 13// will call our constructor.
 14//
 15// The variable name (my_stride_reg) is arbitrary — it just needs to be a
 16// global so it runs at static-initialization time.
 17//
 18// Breaking down the type:
 19//   champsim::modules::prefetcher           — the interface we implement
 20//   ::register_module<my_stride>            — template helper on module_base
 21//   my_stride_reg("my_stride")             — constructor arg = model name
 22// =========================================================================
 23champsim::modules::prefetcher::register_module<my_stride>
 24    my_stride_reg("my_stride");
 25
 26// =========================================================================
 27// CONSTRUCTOR
 28//
 29// The ModuleBuilder is pre-populated by the config system before we see it.
 30// It contains:
 31//   - A pointer to the parent module (a cache_module*)
 32//   - All JSON parameters as a string->std::any map
 33//   - The module name and model name
 34//
 35// get_parameter<T>(name, optional, default):
 36//   - name:     JSON key to look up
 37//   - optional: if true, return default_value when key is missing;
 38//               if false, exit with an error when key is missing
 39//   - default:  value to use when key is missing (only if optional=true)
 40//
 41// get_parent<T>():
 42//   - Returns a T* to the parent module.  For prefetchers and replacement
 43//     policies, T is cache_module.  For branch predictors and BTBs, T is
 44//     core_module.
 45// =========================================================================
 46my_stride::my_stride(champsim::modules::ModuleBuilder builder)
 47    : degree(builder.get_parameter<int>("degree", true, 3)),
 48      cache_(builder.get_parent<champsim::modules::cache_module>())
 49{
 50    // At this point:
 51    //   degree == the "degree" value from JSON, or 3 if absent
 52    //   cache_ == pointer to the parent cache (never null if config is valid)
 53}
 54
 55// =========================================================================
 56// MAIN HOOK: prefetcher_cache_operate
 57//
 58// Called by the cache on every access (loads, stores, prefetch fills).
 59//
 60// Parameters:
 61//   addr           — full byte address of the access
 62//   ip             — instruction pointer that caused the access
 63//   cache_hit      — true if the access was a cache hit
 64//   useful_prefetch— true if a prior prefetch for this address was used
 65//   type           — LOAD, RFO (store), PREFETCH, WRITE, or TRANSLATION
 66//   metadata_in    — metadata propagated from a prior prefetcher in the chain
 67//
 68// Returns: metadata to propagate to the next prefetcher in the chain.
 69// =========================================================================
 70uint32_t my_stride::prefetcher_cache_operate(
 71    champsim::address addr, champsim::address ip,
 72    bool cache_hit, bool useful_prefetch, access_type type,
 73    uint32_t metadata_in)
 74{
 75    // Convert the byte address to a cache-line (block) number.
 76    // champsim::block_number strips the low bits (the block offset),
 77    // giving us a cache-line-granularity address for stride computation.
 78    champsim::block_number cl_addr{addr};
 79
 80    // Look up whether we've seen this IP before.
 81    auto it = table.find(ip);
 82    if (it != table.end()) {
 83        // We've seen this IP before.  Compute the stride: the difference
 84        // between the current cache-line and the last one this IP accessed.
 85        // champsim::offset(a, b) returns b - a in block-number units.
 86        auto stride = champsim::offset(it->second.last_addr, cl_addr);
 87
 88        // Only prefetch if:
 89        //   1. stride != 0   — the IP didn't access the same line twice
 90        //   2. stride == last_stride — the pattern is consistent
 91        if (stride != 0 && stride == it->second.stride) {
 92            for (int i = 0; i < degree; ++i) {
 93                // Compute the prefetch address: current line + (stride * (i+1))
 94                champsim::address pf_addr{cl_addr + (stride * (i + 1))};
 95
 96                // Decide whether to fill this level or a lower level.
 97                // When MSHRs are less than 50% full, fill this cache level
 98                // (L2C).  When they're busy, fill the level below (LLC)
 99                // to avoid polluting the closer cache.
100                bool fill_this_level = cache_->get_mshr_occupancy_ratio() < 0.5;
101
102                // prefetch_line() is provided by the prefetcher base class.
103                // It issues the prefetch through the parent cache.
104                // Arguments: address, fill_this_level, metadata.
105                prefetch_line(pf_addr, fill_this_level, metadata_in);
106            }
107        }
108
109        // Update the tracker for next time.
110        it->second.stride = stride;
111        it->second.last_addr = cl_addr;
112    } else {
113        // First time seeing this IP.  Create a tracker entry with stride 0
114        // (we need at least two accesses to compute a stride).
115        table[ip] = {cl_addr, 0};
116    }
117
118    // Pass metadata through unchanged.
119    return metadata_in;
120}

Step 4: Build¶

Rebuild ChampSim. The Makefile automatically discovers source files under prefetcher/, replacement/, branch/, and btb/:

make -j$(nproc)

If you get a linker error about duplicate symbols, make sure your register_module variable name is unique across all modules.

Step 5: Configure and Run¶

Legacy config — the simplest option, just name the model:

{
    "L2C": { "prefetcher": "my_stride" }
}

To pass parameters to the prefetcher, use the object form (see Passing Parameters to Submodules):

{
    "L2C": { "prefetcher": {"model": "my_stride", "degree": 4} }
}

Explicit config — the prefetcher is a child of the cache module:

{
    "name": "cpu0_L2C_prefetcher",
    "module": "prefetcher",
    "model": "my_stride",
    "degree": 4
}

Run:

bin/champsim --config my_config.json \
    --warmup-instructions 200000000 \
    --simulation-instructions 500000000 \
    trace.champsimtrace.xz

Verify with --dump — add --dump to see every module’s parameters and connections, confirming your prefetcher was constructed with the right degree:

bin/champsim --config my_config.json --dump

Part 2: Creating a Replacement Policy¶

This section builds a simple LRU (Least Recently Used) replacement policy. It follows the same three-step pattern as the prefetcher: inherit, override, register.

Header (replacement/my_lru/my_lru.h):

 1#ifndef MY_LRU_H
 2#define MY_LRU_H
 3
 4#include <vector>
 5#include "modules.h"
 6
 7// -------------------------------------------------------------------------
 8// my_lru — Least Recently Used replacement policy.
 9//
10// Inherits from champsim::modules::replacement, which is
11// module_base<replacement, cache_module>.  Like prefetchers, replacement
12// policies are children of a cache_module.
13//
14// The replacement interface has four pure virtual methods:
15//   initialize_replacement()       — one-time setup
16//   find_victim(...)               — pick a way to evict
17//   update_replacement_state(...)  — called on every access (hit or miss)
18//   replacement_cache_fill(...)    — called when a new line is filled
19//   replacement_final_stats()      — end-of-simulation hook
20// -------------------------------------------------------------------------
21struct my_lru : public champsim::modules::replacement {
22
23    // Number of ways per set — we read this from the parent cache at
24    // construction time so our flat vector is correctly sized.
25    long num_ways;
26
27    // Flat vector of last-used timestamps, indexed as [set * num_ways + way].
28    // A flat vector is faster than a map<pair<long,long>, ...> because the
29    // access pattern is dense and predictable.
30    std::vector<uint64_t> last_used;
31
32    // Monotonically increasing timestamp.  We increment it on every access,
33    // giving us a total ordering of recency.
34    uint64_t cycle = 0;
35
36    // Constructor — reads cache geometry from the parent.
37    my_lru(champsim::modules::ModuleBuilder builder);
38
39    // --- Interface overrides ---
40
41    void initialize_replacement() override {}
42
43    // find_victim: Return the way index within "set" that should be evicted.
44    // The cache calls this when it needs to make room for a new line.
45    long find_victim(uint32_t triggering_cpu, uint64_t instr_id, long set,
46        const champsim::cache_block* current_set, champsim::address ip,
47        champsim::address full_addr, access_type type) override;
48
49    // update_replacement_state: Called on every cache access (hit or miss).
50    // We use it to update the recency timestamp.
51    void update_replacement_state(uint32_t triggering_cpu, long set, long way,
52        champsim::address full_addr, champsim::address ip,
53        champsim::address victim_addr, access_type type, bool hit) override;
54
55    // replacement_cache_fill: Called when a new line is physically written
56    // into the cache.  We also update the timestamp here.
57    void replacement_cache_fill(uint32_t triggering_cpu, long set, long way,
58        champsim::address full_addr, champsim::address ip,
59        champsim::address victim_addr, access_type type) override;
60
61    void replacement_final_stats() override {}
62};
63
64#endif

Implementation (replacement/my_lru/my_lru.cc):

 1#include "my_lru.h"
 2#include <algorithm>
 3
 4// Register under the name "my_lru".
 5champsim::modules::replacement::register_module<my_lru>
 6    my_lru_reg("my_lru");
 7
 8// -------------------------------------------------------------------------
 9// Constructor
10//
11// We query the parent cache for its geometry (num_sets, num_ways) so we can
12// size our data structures correctly.  This is the typical pattern for
13// replacement policies: they need to know how big the cache is.
14// -------------------------------------------------------------------------
15my_lru::my_lru(champsim::modules::ModuleBuilder builder)
16    : num_ways(static_cast<long>(
17          builder.get_parent<champsim::modules::cache_module>()->num_ways())),
18      last_used(static_cast<std::size_t>(
19          builder.get_parent<champsim::modules::cache_module>()->num_sets()
20          * num_ways), 0)
21{
22    // last_used is now a flat vector of size (sets * ways), all initialized
23    // to 0.  Way indices within each set are at offsets [set*ways .. set*ways+ways).
24}
25
26// -------------------------------------------------------------------------
27// find_victim: Scan the last_used timestamps for this set and return the
28// way with the smallest (oldest) timestamp.
29// -------------------------------------------------------------------------
30long my_lru::find_victim(uint32_t /*triggering_cpu*/, uint64_t /*instr_id*/,
31    long set, const champsim::cache_block* /*current_set*/,
32    champsim::address /*ip*/, champsim::address /*full_addr*/,
33    access_type /*type*/)
34{
35    // Point to the start of this set's slice of the flat vector.
36    auto begin = std::next(last_used.begin(), set * num_ways);
37    auto end = std::next(begin, num_ways);
38
39    // std::min_element finds the way with the smallest last-used time.
40    auto victim = std::min_element(begin, end);
41    return std::distance(begin, victim);
42}
43
44// -------------------------------------------------------------------------
45// update_replacement_state: Stamp the accessed way with the current time.
46//
47// We skip writeback hits because writebacks are not demand accesses — they
48// are cache maintenance operations.  Promoting a line on a writeback would
49// artificially extend its lifetime.
50// -------------------------------------------------------------------------
51void my_lru::update_replacement_state(uint32_t /*triggering_cpu*/, long set,
52    long way, champsim::address /*full_addr*/, champsim::address /*ip*/,
53    champsim::address /*victim_addr*/, access_type type, bool hit)
54{
55    if (hit && type != access_type::WRITE)
56        last_used.at(static_cast<std::size_t>(set * num_ways + way)) = cycle++;
57}
58
59// -------------------------------------------------------------------------
60// replacement_cache_fill: A new line was just written into set/way.
61// Mark it as recently used.
62// -------------------------------------------------------------------------
63void my_lru::replacement_cache_fill(uint32_t /*triggering_cpu*/, long set,
64    long way, champsim::address /*full_addr*/, champsim::address /*ip*/,
65    champsim::address /*victim_addr*/, access_type /*type*/)
66{
67    last_used.at(static_cast<std::size_t>(set * num_ways + way)) = cycle++;
68}

Part 3: Creating a Branch Predictor¶

Branch predictors follow the same pattern, but they are children of core_module instead of cache_module.

The branch_predictor interface has three pure virtual methods:

  • initialize_branch_predictor() — one-time setup

  • predict_branch(ip, predicted_target, always_taken, branch_type) — return true for taken, false for not-taken

  • last_branch_result(ip, target, taken, branch_type) — called after the branch is resolved so you can update predictor state

A simple bimodal predictor (branch/my_bimodal/my_bimodal.h):

 1#ifndef MY_BIMODAL_H
 2#define MY_BIMODAL_H
 3
 4#include <vector>
 5#include "modules.h"
 6
 7// -------------------------------------------------------------------------
 8// my_bimodal — A simple bimodal branch predictor.
 9//
10// Uses a table of 2-bit saturating counters indexed by the low bits of the
11// instruction pointer.  This is the simplest dynamic branch predictor.
12//
13// Inherits from champsim::modules::branch_predictor, which is
14// module_base<branch_predictor, core_module>.  Branch predictors are
15// children of a core_module.
16// -------------------------------------------------------------------------
17struct my_bimodal : public champsim::modules::branch_predictor {
18
19    // Number of index bits — controls table size (2^index_bits entries).
20    int index_bits;
21
22    // Table of 2-bit saturating counters.  Each entry is 0–3:
23    //   0,1 = predict not-taken
24    //   2,3 = predict taken
25    // We use int8_t to save space.
26    std::vector<int8_t> table;
27
28    // Constructor reads table_size from JSON config.
29    my_bimodal(champsim::modules::ModuleBuilder builder);
30
31    void initialize_branch_predictor() override {}
32
33    // predict_branch: Return true (taken) or false (not-taken).
34    bool predict_branch(champsim::address ip, champsim::address predicted_target,
35        bool always_taken, uint8_t branch_type) override;
36
37    // last_branch_result: Update the counter after the branch is resolved.
38    void last_branch_result(champsim::address ip, champsim::address target,
39        bool taken, uint8_t branch_type) override;
40};
41
42#endif

Implementation (branch/my_bimodal/my_bimodal.cc):

 1#include "my_bimodal.h"
 2
 3// Register as "my_bimodal".
 4champsim::modules::branch_predictor::register_module<my_bimodal>
 5    my_bimodal_reg("my_bimodal");
 6
 7// -------------------------------------------------------------------------
 8// Constructor
 9//
10// Read table_size from JSON (default 4096 entries = 12 index bits).
11// Initialize all counters to 2 (weakly taken), a common starting point.
12// -------------------------------------------------------------------------
13my_bimodal::my_bimodal(champsim::modules::ModuleBuilder builder)
14    : index_bits(builder.get_parameter<int>("index_bits", true, 12)),
15      table(1 << index_bits, 2)  // 2^index_bits entries, init to "weakly taken"
16{
17}
18
19// -------------------------------------------------------------------------
20// predict_branch
21//
22// Index into the counter table using the low bits of the IP.
23// Counter >= 2 means "predict taken".
24// -------------------------------------------------------------------------
25bool my_bimodal::predict_branch(champsim::address ip,
26    champsim::address /*predicted_target*/, bool /*always_taken*/,
27    uint8_t /*branch_type*/)
28{
29    // Use ip.to<std::size_t>() to extract the raw integer value.
30    // Mask with (table_size - 1) to get an index.
31    std::size_t index = ip.to<std::size_t>() & ((1u << index_bits) - 1);
32    return table[index] >= 2;
33}
34
35// -------------------------------------------------------------------------
36// last_branch_result
37//
38// Increment the counter if the branch was taken, decrement if not-taken.
39// Saturate at 0 and 3 (2-bit counter).
40// -------------------------------------------------------------------------
41void my_bimodal::last_branch_result(champsim::address ip,
42    champsim::address /*target*/, bool taken, uint8_t /*branch_type*/)
43{
44    std::size_t index = ip.to<std::size_t>() & ((1u << index_bits) - 1);
45    if (taken && table[index] < 3)
46        table[index]++;
47    else if (!taken && table[index] > 0)
48        table[index]--;
49}

Part 4: Understanding ModuleBuilder¶

ModuleBuilder is the object the module system passes to your constructor. It carries all the information your module needs to initialize itself.

Reading Parameters¶

get_parameter<T>(name, optional, default_value) reads a JSON value:

// REQUIRED: exits with an error if "sets" is not in the JSON.
// The third argument is ignored when optional=false, but you must provide it
// due to the template signature.
int sets = builder.get_parameter<int>("sets", false, 0);

// OPTIONAL: returns 3 if "degree" is not in the JSON.
int degree = builder.get_parameter<int>("degree", true, 3);

// STRING parameter:
auto mode = builder.get_parameter<std::string>("mode", true, "normal");

// SIZE parameter — uses std::size_t, which is 64-bit unsigned.
// numeric_any_cast handles converting JSON integers (stored as int64_t)
// to std::size_t automatically.
auto sz = builder.get_parameter<std::size_t>("table_size", true, 256);

The parameter name corresponds to a key in the module’s JSON config entry. See Passing Parameters to Submodules and the Explicit Configuration Format for how these values get into the ModuleBuilder.

Accessing the Parent Module¶

get_parent<T>() returns a pointer to the module that owns this submodule:

// Prefetchers and replacement policies → parent is cache_module
auto* cache = builder.get_parent<champsim::modules::cache_module>();

// Branch predictors and BTBs → parent is core_module
auto* core = builder.get_parent<champsim::modules::core_module>();

Store the pointer in a member variable if you need it after construction. Methods available on cache_module include:

  • get_mshr_occupancy_ratio() — MSHR load as a fraction (0.0 to 1.0)

  • get_mshr_size() — total MSHR capacity

  • num_sets() / num_ways() — cache geometry

  • is_virtual_prefetch() — whether prefetch addresses are virtual

  • get_rq_occupancy() / get_wq_occupancy() / get_pq_occupancy() — queue loads

Querying Module Name and Model¶

// The instance name (e.g. "cpu0_L2C_prefetcher"):
std::string name = builder.get_name();

// The model name (e.g. "my_stride"):
std::string model = builder.get_model();

Part 5: Debugging Module Construction¶

Use --dump to see every module that was constructed, its parameters, and its connections:

bin/champsim --config my_config.json --dump

Example output:

[cpu0_L2C_prefetcher] degree = 4 (set)
[cpu0_L2C_prefetcher] created_module = my_stride (set)
[cpu0_L2C_replacement] created_module = lru (set)

This is invaluable for verifying that:

  • Your module was instantiated (look for created_module = your_model_name)

  • Parameters have the expected values ((set) vs (default))

  • Parent/child connections are correct

Part 6: Defining a New Interface¶

ChampSim ships with built-in interfaces for prefetchers, replacement policies, branch predictors, and BTBs. The module system is fully extensible — you can define your own interface types, attach them to existing parent modules, and the config system will construct them for you from JSON.

This section walks through adding a cache_indexer interface: a submodule of a cache that owns the address → set-index mapping. The default mapping in ChampSim is a simple bit-slice of the block address, which is a common source of conflict misses on pathological access patterns (power-of-two strides, aliasing page tables). A pluggable indexer lets you swap the mapping (bit-slice, XOR hashing, prime modulo, skewed associativity) at runtime from the config file without touching the cache core.

The end state is:

  • an interface header inc/cache_indexer.h,

  • an interface registration in src/modules.cc,

  • a hook inside CACHE::get_set_index that consults the indexer when one is attached,

  • two implementations (bit_slice and xor_hash) under indexer/,

  • and a JSON config entry that attaches the indexer as a child of a cache module.

Step 1: Define the Interface¶

Create inc/cache_indexer.h:

 1#ifndef CHAMPSIM_CACHE_INDEXER_H
 2#define CHAMPSIM_CACHE_INDEXER_H
 3
 4#include "modules.h"
 5
 6namespace champsim::modules {
 7
 8// -------------------------------------------------------------------------
 9// cache_indexer — A submodule of cache_module that computes the set index
10// for a given address.
11//
12// Inherits from module_base<cache_indexer, cache_module>.
13//
14// The first template argument is the interface type itself (CRTP).  This
15// keeps the static module_map / instance_map unique to this interface so a
16// cache_indexer registered as "bit_slice" won't collide with a prefetcher
17// registered as "bit_slice".  The second argument is the parent type —
18// because indexers live under a cache, that's cache_module.
19//
20// module_base provides (inherited):
21//   - register_module<T>   : register an implementation by model name
22//   - register_interface   : register this interface type by string name
23//   - create_instance(...) : factory used by the config system
24// -------------------------------------------------------------------------
25struct cache_indexer : public module_base<cache_indexer, cache_module> {
26
27  virtual ~cache_indexer() = default;
28
29  // compute_set_index — map a byte address to a set index in
30  // [0, num_sets()).  The cache asserts the return value is in-range.
31  //
32  // If you want the cache-line-granularity address, strip
33  // get_parent<cache_module>()->get_offset_bits() bits off the bottom.
34  virtual long compute_set_index(champsim::address addr) = 0;
35};
36
37} // namespace champsim::modules
38
39#endif

Why CRTP? The first template parameter to module_base is always the interface class itself. module_base<cache_indexer, cache_module> instantiates a set of static registries that are unique to the cache_indexer interface. A module registered as "lru" under replacement will not be visible through cache_indexer’s registry and vice-versa.

Step 2: Register the Interface¶

The explicit config parser looks up interfaces by string name so that "module": "cache_indexer" in JSON resolves to our module_base specialization. Add a single line to src/modules.cc (where the other interfaces — prefetcher, replacement, branch, btb — are registered):

#include "cache_indexer.h"

static champsim::modules::cache_indexer::register_interface
    cache_indexer_iface_reg("cache_indexer");

This runs at static-initialization time and makes interface_registry::create("cache_indexer", builder, parent) available to the explicit environment. The string "cache_indexer" is exactly what you will use as the "module" key in the JSON config.

Step 3: Hook the Interface into the Parent¶

An interface with no consumer does nothing. The cache needs to construct the indexer submodule from its builder and consult it on every set lookup. Two surgical edits to CACHE are required.

First, in inc/cache.h, store an indexer pointer and populate it in the cache constructor from the builder’s submodules. We treat the indexer as a required submodule: every cache must declare at least one cache_indexer child in its JSON config.

// In the CACHE class body, alongside pref_module_pimpl / repl_module_pimpl:
champsim::modules::cache_indexer* indexer_module_pimpl = nullptr;

// In the CACHE constructor, alongside the existing submodule loops.
// get_submodules defaults to optional=false, so a missing
// "cache_indexer" child causes the simulator to exit with an error
// rather than silently producing an unindexed cache.
for (const auto& sub : builder.get_submodules("cache_indexer"))
  indexer_module_pimpl = champsim::modules::cache_indexer::create_instance(
      sub, static_cast<champsim::modules::cache_module*>(this));

The loop form is intentional. get_submodules returns a vector; writing it as a loop handles any-N submodules uniformly. For this interface we keep only the last one because a cache has a single address → set mapping; another interface (e.g. a listener) would push_back into a vector here instead. If you wanted to permit caches without an indexer, you would pass optional=true as the second argument — get_submodules would then return an empty vector instead of erroring out, and the loop body would simply not run.

Second, in src/cache.cc, route get_set_index through the indexer:

long CACHE::get_set_index(champsim::address address) const
{
  long idx = indexer_module_pimpl->compute_set_index(address);
  assert(idx >= 0 && static_cast<uint32_t>(idx) < NUM_SET);
  return idx;
}

Step 4: Write Implementations¶

Implementations live under indexer/<model_name>/ so the Makefile can discover them. (Add indexer to the find list in the Makefile’s module_sources line alongside branch btb prefetcher replacement — the build system rules follow the same pattern as the other module directories.)

The default bit-slice, made explicit (indexer/bit_slice/bit_slice.h):

 1#ifndef BIT_SLICE_INDEXER_H
 2#define BIT_SLICE_INDEXER_H
 3
 4#include "cache_indexer.h"
 5#include "champsim.h"
 6
 7struct bit_slice : public champsim::modules::cache_indexer {
 8  long num_sets;
 9  champsim::data::bits offset_bits;
10
11  bit_slice(champsim::modules::ModuleBuilder builder);
12  long compute_set_index(champsim::address addr) override;
13};
14
15#endif

indexer/bit_slice/bit_slice.cc:

 1#include "bit_slice.h"
 2#include "cache.h"
 3
 4// Register under the model name "bit_slice".
 5champsim::modules::cache_indexer::register_module<bit_slice>
 6    bit_slice_reg("bit_slice");
 7
 8bit_slice::bit_slice(champsim::modules::ModuleBuilder builder)
 9  : num_sets(static_cast<long>(
10      builder.get_parent<champsim::modules::cache_module>()->num_sets())),
11    offset_bits(
12      builder.get_parent<champsim::modules::cache_module>()->get_offset_bits())
13{}
14
15long bit_slice::compute_set_index(champsim::address addr)
16{
17  // Canonical bit-slice: use champsim::address::slice with a
18  // dynamic_extent describing [offset_bits, offset_bits + lg2(num_sets)),
19  // then unwrap the strongly-typed slice with .to<long>(). This is the
20  // same expression CACHE::get_set_index uses for its built-in fallback.
21  return addr
22      .slice(champsim::dynamic_extent{offset_bits, champsim::lg2(num_sets)})
23      .to<long>();
24}

A one-line XOR-folding hash that reduces conflict misses on power-of-two strides (indexer/xor_hash/xor_hash.cc):

 1#include "xor_hash.h"
 2#include "cache.h"
 3
 4champsim::modules::cache_indexer::register_module<xor_hash>
 5    xor_hash_reg("xor_hash");
 6
 7xor_hash::xor_hash(champsim::modules::ModuleBuilder builder)
 8  : num_sets(static_cast<long>(
 9      builder.get_parent<champsim::modules::cache_module>()->num_sets())),
10    index_bits(champsim::lg2(num_sets)),
11    offset_bits(
12      builder.get_parent<champsim::modules::cache_module>()->get_offset_bits())
13{}
14
15long xor_hash::compute_set_index(champsim::address addr)
16{
17  // Fold two adjacent index_bits-wide fields of the block address
18  // together. Each field is pulled with champsim::address::slice + a
19  // dynamic_extent whose second argument is the slice width in bits, so
20  // we stay inside the strongly-typed address API.
21  const auto low_bits  = champsim::dynamic_extent{offset_bits, index_bits};
22  const auto high_bits = champsim::dynamic_extent{
23      offset_bits + champsim::data::bits{index_bits}, index_bits};
24  const auto low  = addr.slice(low_bits).to<unsigned long>();
25  const auto high = addr.slice(high_bits).to<unsigned long>();
26  return static_cast<long>(low ^ high);
27}

Note

Prefer champsim::address::slice (combined with champsim::dynamic_extent or one of the named extents in inc/extent.h) over manually converting the address to a uint64_t and shifting. The slice API keeps the offset and width checked by the type system, composes with champsim::address::splice and slice_upper for more complex hashes, and — importantly — is what the rest of ChampSim uses, so your indexer reads like the surrounding code.

Step 5: Configure It¶

In the explicit config, the indexer is a child of a cache module, right next to the prefetcher and replacement submodules:

{
  "name": "LLC", "module": "cache", "model": "DEFAULT_CACHE",
  "num_sets": 2048, "num_ways": 16,
  "children": [
    {"name": "llc_pf",   "module": "prefetcher",    "model": "no"},
    {"name": "llc_repl", "module": "replacement",   "model": "lru"},
    {"name": "llc_idx",  "module": "cache_indexer", "model": "xor_hash"}
  ]
}

Run bin/champsim --config my_config.json --dump and confirm the line:

[llc_idx] created_module = xor_hash (set)

appears before the cache itself is constructed. If the cache_indexer child is omitted from the config, get_submodules("cache_indexer") exits with a required submodules of interface cache_indexer not found error, making the missing wiring impossible to overlook.

Summary¶

Defining a new interface is four pieces:

  1. Interface class inheriting module_base<YourInterface, ParentType>.

  2. Interface registration via register_interface("name") in a .cc file.

  3. Implementation class(es) inheriting the interface, registered with register_module<Impl>("model_name").

  4. Parent integration — the parent module pulls submodule builders out via builder.get_submodules("name"), creates instances with YourInterface::create_instance(sub, this), and calls the virtual methods at the relevant points.

The module_base template provides all the factory and registration infrastructure. You define the virtual methods and the integration point inside the parent module.

Part 7: Migrating from Legacy Modules¶

If you have modules written for an older version of ChampSim that used free-function hooks (e.g. void CACHE::prefetcher_initialize()), here is how to update them:

  1. Create a class inheriting from the appropriate interface (champsim::modules::prefetcher, ::replacement, ::branch_predictor, or ::btb).

  2. Move your free functions into the class as method overrides:

    • Remove the CACHE:: or O3_CPU:: prefix.

    • Update parameter types: uint64_t addresses become champsim::address, uint8_t type becomes access_type type, etc.

    • Add override to each method.

  3. Move global state into member variables. Each module instance gets its own state — you no longer need to key state by cache/CPU pointer.

  4. Add a constructor that takes champsim::modules::ModuleBuilder. If you need access to the parent cache or core, call builder.get_parent<T>().

  5. Add a registration line in your .cc file:

    champsim::modules::prefetcher::register_module<my_pref>("my_pref");

  6. Remove the __legacy__ file if one exists in your module directory.

  7. Stub any new pure virtual methods that didn’t exist in the old interface (e.g. prefetcher_branch_operate, replacement_cache_fill) with override {}.

Example diff for a prefetcher:

// OLD (free-function style):
void CACHE::prefetcher_initialize() { ... }
uint32_t CACHE::prefetcher_cache_operate(uint64_t addr, uint64_t ip, ...) { ... }

// NEW (class-based style):
struct my_pref : champsim::modules::prefetcher {
    my_pref(champsim::modules::ModuleBuilder builder) {}
    void prefetcher_initialize() override { ... }
    uint32_t prefetcher_cache_operate(champsim::address addr, champsim::address ip, ...) override { ... }
    // ... all other pure virtuals stubbed or implemented
};
champsim::modules::prefetcher::register_module<my_pref>("my_pref");

Part 8: Modernizing the Default Config¶

The default champsim_config.json shipped with the repository is a documentation artifact. Its own header says so:

"_description": "Sample/documentation config. Does not represent a
 reasonable system; intended only to illustrate config file structure."

Running experiments against it and reporting the results as representative of a contemporary CPU is a common source of bad numbers. The defaults were reasonable in the mid-2010s; they are not in 2026. This section walks through upgrading the default config to something closer to an Intel Redwood Cove (Meteor Lake / Granite Rapids P-core) system and explains what each change buys you. The approximate specs we are targeting:

  • ~5 GHz core clock, ~6-wide front-end, ~6-wide allocate, 12-wide (MATH + LOAD + STORE) execute.

  • 512-entry ROB, ~280-entry INT / ~332-entry FP register files, ~205-entry scheduler, ~192-entry load queue, ~114-entry store queue.

  • 48 KB 12-way L1D, ~5 cycle load-to-use at 5 GHz.

  • 64 KB 8-way L1I (doubled vs. Golden Cove — one of Redwood Cove’s visible front-end changes).

  • 2 MB private L2 (client parts; Granite Rapids doubles this to 4 MB).

  • ~3 MB / core 12-way shared L3 slice (client).

  • DDR5-5600, 2 channels (client) — Granite Rapids uses DDR5-6400 / MCR-DIMM.

The goal here is not perfect fidelity (ChampSim’s core model is abstract, and does not model a specific ISA or microarchitecture) but realistic values so that replacement/prefetch comparisons are not dominated by obvious model artifacts (e.g. a ROB that empties in a single cycle).

Where to Direct Attention¶

When cloning champsim_config.json into an experimental config, audit these parameters in roughly decreasing order of impact:

  1. Core width and window sizes (rob_size, lq_size, sq_size, scheduler_size, *_width). The default ROB of 352 and LQ of 128 are usably close to Redwood Cove, but the 4-wide execute / 2-wide LQ are too narrow for a modern OoO core and silently gate MLP.

  2. L1D latency and size. The default 5/64/12 is fine; double-check it matches the accuracy you want.

  3. L2 size. The default 512 KB / 8-way L2 is ~4× smaller than Redwood Cove’s 2 MB / 16-way private L2 (client parts). Undersized L2s push more traffic to LLC and DRAM and distort prefetcher evaluations.

  4. LLC size. The default 2048 sets × 16 ways × 64 B = 2 MB. Modern server parts are 30 MB+; client parts are 30 MB+. Per-core LLC slices are in the 1.5–3 MB range. Running with 2 MB shared LLC for any benchmark with a working set larger than ~1 MB is a DRAM-bound experiment by construction.

  5. DRAM speed. The default data_rate: 3200 (DDR4-3200) is two generations behind the DDR5-5600 (client) / DDR5-6400 + MCR-DIMM (server) that ships with Redwood Cove systems.

  6. Prefetchers. The defaults are no at L1D/L1I and spp_dev at L2C. This asymmetry is a deliberate documentation choice; for a realistic comparison baseline you typically want ip_stride or va_ampm_lite at L1D (both are virtually-addressed L1D prefetchers in this repo — va_ampm_lite is page-region AMPM on VA), next_line at L1I, and spp_dev at L2C.

  7. Memory timings (nCAS, nRCD, nRP, nRAS). The defaults target DDR4; DDR5 timings are different in absolute cycles.

  8. MSHR sizes. L1D MSHR = 16 is acceptable; Redwood Cove sustains ~16 outstanding demand misses. However, L2C MSHR = 32 is undersized. Redwood Cove has 64 MSHRs in its L2C. An undersized MSHR caps effective MLP.

"L1I": { "sets": 128, "ways": 8,  "mshr_size": 16, "latency": 3,
         "prefetcher": "next_line" },

Sizes: 64 KB / 48 KB / 2 MB / ~3 MB (per core) with 64 B lines. Latencies are in cache cycles. Note that ChampSim’s single-LLC model does not distinguish slice access time from interconnect hops; 40 is the aggregate LLC hit latency in LLC cycles.

The defaults omit replacement at all levels except LLC; for a modern baseline, srrip at LLC is a reasonable starting point. Leave L1/L2 at the built-in LRU by omitting the key.

Step 6: Why the Defaults Are What They Are¶

The shipped defaults are optimized for readability of the config file, not realism. Small numbers (8-way, 64-set) make it obvious which field is which when you’re learning the format.

When in doubt, start from champsim_config.json for structure, refer to published values for your config, and --dump after every edit to confirm the settings landed where you expected.
